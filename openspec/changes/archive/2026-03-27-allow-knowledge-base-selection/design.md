## Context

当前代码已经具备“显式选中来源列表”这条主线，但这条能力被人为收窄到了文档节点：首页树只允许勾选 `document`，文档子树覆盖判断也只针对 `document`，而 `folder` 仍然只能退回到单一 `selectedScope` 入口。结果是目录型内容组织在 UI 上不能加入多选集合，任务摘要也无法表达“目录 + 文档”这类混合来源。

这次需求会影响首页树节点可勾选策略、来源归一化与覆盖关系判断、任务摘要生成，以及测试样例中的知识库树行为，因此需要先把“哪些节点可以作为显式来源根”定义清楚。

## Goals / Non-Goals

**Goals:**

- 支持同一知识库内的目录/库节点与文档节点一起作为显式同步来源。
- 让目录来源像文档子树来源一样参与覆盖判断，避免祖先目录与后代文档重复入队。
- 保持整库同步、单目录同步、单文档同步和已有文档多选任务的兼容性。
- 在首页摘要和任务历史中显示可信的混合来源类型与有效文档数量。

**Non-Goals:**

- 不支持跨多个知识库混合勾选目录或文档后创建一个任务。
- 不把多维表格等非文档叶子节点改成可同步来源。
- 不调整同步管线里的内容抓取、渲染、图片处理或鉴权策略。

## Decisions

### 1. 将“显式来源根”泛化为 folder/document，而不是继续只认 document

首页树的 checkbox 规则调整为：`folder` 和 `document` 都可以勾选，`space` 仍然保持单范围入口，`bitable` 继续不可选。这样可以让“库/目录”进入现有 `selectedSources` 契约，而不必重新设计任务创建接口。

之所以不把 `space` 一并塞进多选集合，是因为整库同步已经有明确单范围语义；把它也混入 checkbox 会让“整库覆盖目录/文档”的优先级更复杂，超出这次需求。

备选方案：

- 把目录点击事件直接转换成单一 `selectedScope`，继续不进入多选集合。否决原因是无法满足“目录和文档组合选择”的真实诉求。

### 2. 用统一的祖先覆盖规则归一化 folder/document 混合来源

来源归一化不再只针对文档子树，而是基于同一知识库内的路径前缀关系判断覆盖：

- 选中的 `folder` 覆盖它下面的所有 `folder`/`document` 来源。
- 选中的“含子文档”的 `document` 覆盖它下面的后代 `document` 来源。
- 普通叶子文档只覆盖自己。

归一化后优先保留更高层的祖先来源，避免同一文档通过“目录 + 子文档”或“父文档 + 孙文档”重复进入规划集合。

备选方案：

- 仅做 documentId 级别去重，不处理来源级覆盖。否决原因是任务摘要会保留冗余来源，用户会看到一组互相包含的选择结果。

### 3. 继续限制显式多选只发生在同一知识库内

新增 folder 勾选后仍保持“单次任务只能来自同一 knowledge base”的校验；如果用户在另一知识库继续勾选，现有集合会被切换到当前知识库。这样可以复用已有输出摘要、任务上下文和权限提示逻辑，避免本次把问题扩展成跨库组合同步。

备选方案：

- 允许跨库混选并在任务层记录多个知识库。否决原因是会同时扩大摘要表达、路径镜像、权限失败提示等多个面向。

### 4. 摘要类型从“多文档”扩展为“混合来源集合”

`buildSelectionSummary()` 需要识别三类显式来源集合：单目录/单文档/多来源集合。多来源集合里如果包含 `folder`，展示文案应突出“目录和文档混合”或“多个同步根”，而不是继续写成“多篇文档”。有效文档数仍以规划前解析出的文档总数为准。

这样可以让首页和任务历史对用户的原始意图保持可解释，不会在选择了一个目录后仍显示成“多文档同步”。

## Risks / Trade-offs

- [目录来源加入 checkbox 后，树上可选节点变多，用户可能更难区分“点击选中范围”和“勾选加入集合”] → 保持现有点击选择 `selectedScope` 行为，同时用摘要区和提示语明确当前生效的是显式来源集合。
- [祖先覆盖规则泛化后，前端禁用态和后端归一化若不一致会产生错觉] → 把覆盖判断抽成共享工具函数，并用测试同时覆盖 UI 禁用态和任务归一化结果。
- [混合来源摘要如果过度详细会让任务列表噪声过大] → 默认展示类型、知识库名、有效文档数和少量路径预览，避免完整展开所有来源路径。
