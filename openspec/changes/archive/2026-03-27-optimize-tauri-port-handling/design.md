## Context

当前项目在三个位置依赖固定本地端口：Tauri 开发态 `devUrl` 使用 `1430`，Vite HMR 使用 `1431`，桌面 OAuth 回调仅尝试 `3000` 和 `3001`。这让本地开发和用户授权都对环境非常敏感，只要端口被其他程序占用就会直接失败。项目已经使用 Tauri OAuth 插件提供可选端口列表，因此更稳妥的方向是扩大候选端口池、放宽开发态固定端口约束，并把配置说明与失败提示同步到 UI/README。

## Goals / Non-Goals

**Goals:**
- 让 `tauri dev` 在默认开发端口被占用时仍能自动找到可用端口启动
- 让桌面 OAuth 回调在常见本地端口冲突场景下仍有较高概率成功初始化
- 在端口资源全部不可用时向用户展示可操作的提示，而不是笼统失败
- 让设置文案和开发文档与新的端口策略保持一致

**Non-Goals:**
- 不引入长期运行的本地 HTTP 服务
- 不修改 Feishu OAuth 协议本身或回调 URI 结构
- 不实现任意随机端口并动态回写到飞书开放平台配置，因为飞书回调地址仍需预先登记

## Decisions

### 1. Vite 开发服务器改为允许端口回退

将 `vite.config.ts` 中的 `strictPort` 从强制固定改为允许回退，并移除固定 HMR 端口。这样在 `1430` 被占用时，Vite 可以自动尝试下一个可用端口，Tauri CLI 通过 dev server 实际启动结果连接前端。

备选方案：
- 继续固定端口并要求开发者手工排障：成本低，但体验差
- 启动前自行扫描并写回 `tauri.conf.json`：更复杂，也不如直接交给 Vite 处理

### 2. OAuth 回调使用受控端口池而非单一或双端口

保留 `localhost` 回调模式，但把候选端口从 `[3000, 3001]` 扩展为一组连续端口，例如 `3000-3010`。这兼顾了两点：一是用户可以在飞书应用里一次性登记有限范围的回调地址，二是冲突时仍有多次重试空间。

备选方案：
- 使用 `127.0.0.1:0` 随机端口：技术上最灵活，但飞书端无法预先登记无限随机回调地址
- 维持 2 个端口：实现最简单，但对真实环境冲突过于脆弱

### 3. 对端口耗尽给出显式错误与配置指引

授权页在初始化本地 OAuth 回调失败时，应提示用户检查端口占用，且设置页/README 应列出完整的回调地址范围，减少“代码支持多个端口，但文档只写两个端口”的不一致。

备选方案：
- 只改代码，不改文档：会让用户难以正确配置飞书应用
- 只改文档，不改错误提示：出问题时仍然难排查

## Risks / Trade-offs

- [回调地址数量变多] -> 在飞书开放平台需要登记更多 `localhost` 地址；通过限制在较小连续范围内降低维护成本
- [开发端口不再固定] -> 某些依赖固定端口的本地脚本可能需要适配；通过 README 明确说明 Vite 可能回退到其他端口
- [HMR 使用默认端口策略] -> 个别网络环境下 WebSocket 配置可能和当前定制逻辑不同；保留 `TAURI_DEV_HOST` 下的 host 指定，仅去掉不必要的固定 HMR 端口
