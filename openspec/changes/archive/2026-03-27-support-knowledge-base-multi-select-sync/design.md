## Context

当前前后端都把同步来源建模为单个 `selectedScope`。首页树控件使用单选模式，创建任务时只提交一个 scope，任务记录和同步规划也只围绕该单个范围展开。这与参考项目中“在同一知识库内勾选多篇文档后一次性执行”的使用方式不一致，导致用户只能扩大到目录/整库同步，或者反复为多个离散文档创建任务。

这次变化会同时影响前端树选择状态、任务创建入参、后端文档发现逻辑，以及任务历史中的来源展示，因此需要先定义统一的数据模型和约束。

## Goals / Non-Goals

**Goals:**

- 支持在同一知识库内显式多选多个文档节点并创建一次同步任务。
- 支持对带有子文档的父文档执行一键选择，将父文档及其全部后代文档一次加入当前多选集合。
- 让同步规划基于“选中文档集合”构建去重后的队列，而不是仅依赖单个 scope。
- 在首页摘要和任务历史中展示可信的多选来源信息，包括所属知识库与选中文档数量。
- 保持整库同步、目录同步、单文档同步继续可用，并让旧任务记录仍可展示。

**Non-Goals:**

- 不支持跨多个知识库组合文档后一次创建同步任务。
- 不在本次设计中引入“目录 + 单文档”混合勾选规则。
- 不改变 Markdown 渲染、图片处理、权限校验等同步管线的既有行为。

## Decisions

### 1. 用“显式选中来源列表”替代“单一 scope”作为同步规划输入

新增规范化的 `selectedSources` 集合来表示用户本次真正选中的来源项，每个来源项至少包含 `spaceId`、`spaceName`、`documentId`、`nodeToken`、`displayPath` 和 `pathSegments`。对于整库/目录/单文档这类已有入口，创建任务前先归一化成一组来源描述，再交给同步规划层处理。

这样做的原因是多选文档本质上不是单个 scope 的扩展字段，而是“多个离散来源”的组合。继续复用单一 `selectedScope` 会让去重、任务展示和后续重试语义都变得不清晰。

备选方案：

- 继续沿用 `selectedScope`，仅在文档模式下额外挂一个 `selectedDocumentIds` 数组。否决原因是任务历史和同步规划仍然要区分多种分支，模型会越来越难维护。

### 2. 首版多选能力限定为“同一知识库内的文档节点”

树控件改为支持对文档节点进行勾选；知识库和目录节点继续承担导航、懒加载和单 scope 同步入口角色。多选模式下，只允许勾选同一 `spaceId` 下的文档，避免跨知识库组合导致任务上下文、输出摘要和权限提示变得复杂。

这样做可以贴合参考项目中的实际使用方式，并将规则收敛到“显式文档集合”。如果后续要支持跨知识库或目录混选，可以在这个集合模型上继续扩展，而不必重新设计任务契约。

备选方案：

- 允许任意节点都可勾选并在提交时自动展开。否决原因是目录与整库节点会和文档节点发生包含关系冲突，需要额外定义覆盖、去重和展示优先级，超出本次范围。

### 3. 父文档子树选择通过前端递归展开后显式落到文档集合

对于带有子节点的文档，首页树需要提供一个一键入口，把该父文档和所有后代文档一次加入当前多选结果。实现时不新增后端“文档子树”契约，而是由前端递归加载该文档下的后代节点，并把解析出的文档节点显式写入 `selectedSources`。

这样做的原因是当前同步规划已经围绕显式文档集合完成去重、摘要和兼容逻辑。如果为了这个快捷操作新增新的树型来源语义，会再次扩大前后端契约面。

备选方案：

- 在后端新增“父文档及其子文档”专用选择模式。否决原因是会引入新的来源语义、摘要规则和兼容逻辑，而本需求本质上只是多选交互的快捷入口。

### 4. 后端以文档 ID 去重并保留来源摘要

同步任务创建时，后端先把显式选中的文档来源解析为待同步文档集合，再以 `documentId` 去重，避免同一文档因为多次选中或未来扩展入口重复入队。任务记录持久化时同时保存：

- 规划输入：规范化后的 `selectedSources`
- 展示摘要：知识库名、选中文档数量、前若干个显示路径，以及“多选文档”模式标记

这样可以同时满足执行正确性和任务历史可读性。摘要独立保存后，前端无需在列表页反推全部路径即可展示结果。

备选方案：

- 仅保存去重后的文档列表，不保存来源摘要。否决原因是任务列表需要展示用户原始选择语义，否则“为什么只同步了这些文档”会缺乏解释。

### 5. 兼容旧任务记录，采用向后兼容读取

已有任务和 manifest 里可能只有 `selectedScope`。读取历史任务时，如果没有新的 `selectedSources` 或摘要字段，就回退到旧的单 scope 展示逻辑；新任务则写入新的多选字段，并在必要时附带兼容性的单 scope 摘要。

这样可以避免一次性迁移已有本地数据，也降低回滚风险。

## Risks / Trade-offs

- [多选状态与单选入口并存会增加首页交互复杂度] → 通过明确区分“勾选文档用于批量同步”和“点击节点查看/单范围同步”，并在摘要区显示当前模式来降低歧义。
- [任务历史展示过多文档路径会造成噪音] → 默认展示知识库名、文档数量和有限条目预览，其余通过展开区域查看。
- [后端仍需兼容旧字段，短期内模型会更宽] → 以新字段为主、旧字段只读回退，并在后续变更中逐步清理兼容层。
- [只支持同知识库文档多选可能不足以覆盖未来需求] → 先把契约做成“来源列表”，后续若要支持更宽的组合选择只需扩展校验规则而不是推翻模型。
