## Context

当前系统已有 `check_document_freshness` 命令，可以检查文档新鲜度状态，但每次都需要实时查询飞书 API：
- 每次页面加载都会触发 API 调用
- 检查结果没有持久化存储
- 前端无法直接获取已检查过的状态

### 现有架构
- **Manifest**: `sync-manifest.json` 存储同步记录
- **Freshness 检查**: `check_document_freshness` 命令实时查询飞书 API
- **前端**: `HomePage.tsx` 显示同步状态，但没有新鲜度显示

## Goals / Non-Goals

**Goals:**
- 将新鲜度检查结果持久化到 SQLite 数据库
- 提供 CRUD 命令供前端查询和更新
- 在已同步文档后显示新鲜度状态图标
- 支持增量更新，避免全量检查

**Non-Goals:**
- 不改变现有的 `check_document_freshness` 命令逻辑
- 不实现自动后台刷新
- 不实现用户手动触发检查功能

## Decisions

### 1. 使用 SQLite 存储新鲜度元数据

**选择**: SQLite
**备选方案**:
- JSON 文件：读写性能较差，不支持增量更新
- 内嵌在 manifest 中：耦合度高，职责不清

**理由**:
- 项目已有 `rusqlite` 可用
- 支持高效的 CRUD 操作
- 单文件部署，便于迁移

### 2. 数据库位置

**选择**: `{sync_root}/.freshness-metadata.db`
**理由**:
- 与同步数据在同一目录，便于管理
- 用户可以轻松备份或删除

### 3. 前端新鲜度显示

**选择**: 在已同步文档后显示图标指示器
**备选方案**:
- 单独的"检查新鲜度"按钮：增加用户操作步骤
- 全局状态栏：不够直观

**理由**:
- 直接在文档旁显示，信息就近可得
- 使用不同颜色图标区分状态，视觉直观

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| SQLite 数据库损坏 | 数据可重新生成，不影响核心同步功能 |
| 检查 API 频率限制 | 使用 2 秒 debounce，避免频繁调用 |
| 迁移到其他电脑需复制 .db 文件 | 文档说明数据库位置和迁移方法 |

## Migration Plan

1. 部署后首次加载页面时会创建空的数据库
2. 检查已同步文档时自动填充数据
3. 无需手动迁移，自动生成
