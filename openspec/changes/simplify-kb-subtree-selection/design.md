## Context

当前首页知识库树把“点击节点选择同步范围”和“勾选文档做多文档同步”混在一起，但文档节点本身仍按“只选中当前文档”处理，所以才额外补了“一键选中父子文档”按钮。这个交互既比参考工程的树形勾选更绕，也和同步工具的真实使用方式不匹配: 用户通常关心的是某个文档分支整体是否同步，而不是在同一父文档下面精细排除部分子文档。

这次调整会同时影响前端树控件状态、任务创建入参、同步规划时的文档发现，以及任务历史中的来源说明。尤其要解决两个约束:

- 树数据仍然按层级懒加载，父文档被选中时不能依赖“所有子文档都已在前端展开”。
- 现有任务记录和部分实现把多文档集合当作离散文档列表处理，新语义需要兼容旧数据且避免误导用户。

## Goals / Non-Goals

**Goals:**

- 让文档节点默认表示“该文档及其全部后代文档”的同步选择语义。
- 让已被父文档覆盖的后代文档在树上显示为不可单独勾选，去掉额外的“一键选中父子文档”按钮。
- 保持整库同步、目录同步与同知识库内多个文档分支选择仍然可用。
- 让任务创建和历史展示记录“文档分支/子树”语义，而不是仅把结果描述成离散多文档集合。
- 兼容已有旧任务记录与旧的单文档/多文档数据。

**Non-Goals:**

- 不支持跨多个知识库混合选择文档分支。
- 不引入“目录 + 文档分支”混合勾选规则。
- 不改变同步渲染、图片处理、鉴权或输出目录结构。

## Decisions

### 1. 将文档 source 重新定义为“文档子树根节点”

选中一个文档节点时，持久化和任务创建层记录的是“被选中的文档根节点”，其语义是“该节点及其全部后代文档”。后代文档不再作为独立选择结果逐个保存，避免把一个逻辑上的文档分支错误建模为一组偶然凑出来的离散文档。

这样做的原因是用户操作的真实意图是“同步这个分支”，不是“手工列出该分支里的每一篇文档”。把根节点作为选择契约，后续任务历史、摘要与重试也更容易保持一致。

备选方案:

- 继续沿用当前“一键选中后把所有后代文档展开成显式列表”的做法。否决原因是任务历史会继续丢失“这是一个父文档分支”的原始语义，且分支越大，选择状态越难维护。

### 2. 树控件继续使用受控选择，但显式计算“被覆盖后代”禁用态

首页树仍由前端受控 `checkedKeys` 驱动，但勾选集合改为“已选文档根节点集合”。对每个已选根节点，前端根据已加载树节点和必要的按需补载结果，计算其覆盖的后代节点，并将这些后代节点标记为禁用复选框；如果用户切换到另一个知识库的文档分支，则清空之前的跨库文档分支选择。

这样做比直接依赖树组件默认级联勾选更稳妥，因为当前页面还同时存在整库/目录单范围入口、懒加载节点和非文档节点，默认级联行为很难准确表达“可点击浏览但不可单独勾选”的状态。

备选方案:

- 完全切换到树组件原生的父子级联勾选。否决原因是知识库/目录节点并不是这次要开放的多选项，而且懒加载时原生级联很难保证未展开分支的禁用与摘要一致。

### 3. 任务创建与同步规划在提交阶段展开文档子树

创建任务时，前端或后端以“已选文档根节点集合”为输入，对每个根节点递归获取其全部后代文档，构建最终去重后的同步文档队列。对于没有子文档的叶子文档，这个展开结果仅包含自身；对于父文档，则包含根节点与全部后代文档。整库和目录来源继续沿用现有展开逻辑。

这样可以把 UI 的树选择语义和执行层的文档队列解耦: 页面只关心“选了哪些分支”，规划层才关心“最终要同步哪些具体文档”。

备选方案:

- 在前端勾选瞬间就把整棵分支全部展开并永久存成显式文档列表。否决原因是会让懒加载树在交互阶段承担过多发现逻辑，也会放大大分支选择的状态写入成本。

### 4. 任务摘要基于根节点语义生成，并附带已解析文档数量

任务记录继续持久化选择来源，但摘要文字改为优先表达“单个文档分支”或“多个文档分支”，同时在任务创建完成后记录解析得到的实际文档数量和部分路径预览。首页摘要在创建任务前展示根节点路径/分支数量，任务列表则展示解析后的可信文档数量。

这样做既能让用户看到自己选的是哪几个分支，也不会丢失执行时真正会同步多少篇文档的关键信息。

备选方案:

- 只展示根节点路径，不展示解析后的文档数量。否决原因是用户很难预估任务规模。

### 5. 旧任务记录继续按旧字段回退展示

读取历史任务时，如果记录里只有旧的离散 `selectedSources` 或 `selectedScope`，则保留旧的单文档/多文档展示逻辑；新任务写入新的根节点语义摘要字段，并允许从旧字段推导兼容视图。

这样可以避免一次性迁移本地任务数据，同时为后续逐步收敛旧字段保留空间。

## Risks / Trade-offs

- [文档根节点语义需要额外的展开步骤] → 在任务创建阶段统一展开并缓存结果，避免把复杂度散落到多个页面交互分支。
- [懒加载树可能导致禁用态与未加载节点暂时不同步] → 勾选父文档时主动补载该分支所需的最小后代信息，再更新禁用态。
- [新旧任务摘要并存会让数据模型短期更复杂] → 约定新摘要字段优先，旧字段仅保留读取兼容，不再新增依赖。
- [同知识库多分支选择可能出现祖先/后代重叠] → 提交前按祖先覆盖关系去重，优先保留更高层的已选根节点。

## Migration Plan

1. 增加新的文档分支选择与摘要建模，同时保留旧任务读取兼容。
2. 将首页树交互切换到“父文档默认覆盖子树、后代禁用”的行为，并移除独立快捷按钮。
3. 在任务创建/规划阶段统一展开文档分支并做去重。
4. 验证新任务展示正确后，保留旧任务回退逻辑，不做本地数据强制迁移。

## Open Questions

- 当前实现阶段默认把“无后代的文档节点”继续展示为“单个文档”，而“有后代的文档节点”展示为“文档分支”；如果实现中发现统一文案更稳定，可在 apply 阶段一并细化到摘要规则。
